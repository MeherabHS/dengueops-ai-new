# DengueOps AI — Methodology Summary

## P1.1 rolling-origin validation

Primary validation uses an expanding window with 104 initial training rows, a one-row label-availability embargo, one validation row, a two-week target horizon, and weekly origins. P1.3 uses the 68 active Random Forest residuals for a 90%-nominal prior-only expanding absolute-residual empirical range, with 20 warm-up and 48 evaluation folds. Historical empirical coverage is not a probability guarantee or real-world calibration. The legacy RF holdout-RMSE triplet remains separate planning compatibility.

## P1.2A candidate comparison

Seven fixed candidates are compared on the exact P1.1 fold IDs and matrix hashes. Selection uses lowest MAE among candidates completing all 68 folds, followed by declared deterministic tie-breakers. No tuning or statistical-significance claim is made. P1.2B independently validates the current-run P1.2A result and adopts the frozen Random Forest as the active synthetic demonstration model. This does not establish real-world Dhaka superiority.

## 1a. Climate Layer

**Default mode:** Synthetic climate data generated by `analytics/generate_demo_data.py`.
Weekly rainfall, temperature, and humidity for Dhaka South 2024–2026 with a realistic
monsoon seasonal pattern. The 2026 data shows an early monsoon onset (week 20), consistent
with the early dengue pressure warning visible in the case data.

**Climate co-variate epidemiological rationale:**
- Rainfall at 2- and 4-week lag: provisional candidate inputs, not proven or locally selected for Dhaka
- Temperature at 2- and 4-week lag: provisional candidate inputs requiring local temporal backtesting
- Humidity at 2 week lag: Mosquito survival and biting rate modulation

**Optional NASA POWER mode (`--use-nasa-power-climate`):**

When the flag is enabled, `climate_data.csv` is sourced from **NASA POWER**
(Prediction Of Worldwide Energy Resources, NASA Langley Research Center) — publicly
available surface meteorology derived from satellite observations and reanalysis models.

Variables: `PRECTOTCORR` (rainfall sum), `T2M` (mean temperature), `RH2M` (mean humidity).
Coordinates: Latitude 23.7104, Longitude 90.4074 (Dhaka South centroid).

```bash
python analytics/run_pipeline.py --use-nasa-power-climate
```

## 3. Forecast Model

**Model:** `sklearn.ensemble.GradientBoostingRegressor`

**Rationale:**
- Handles non-linear interactions between climate and case features
- Feature importance is currently unavailable until generated from the fitted model
- Robust in low-sample settings vs deep learning
- No attention-based or sequence model complexity needed at city-level weekly granularity

**Hyperparameters (Phase 0 defaults):**
- n_estimators: 300
- learning_rate: 0.05
- max_depth: 4
- subsample: 0.8

---

## 4. Validation

**Strategy:** Walk-forward (expanding window) time-based split

- No future data leakage
- Each test prediction uses only data available at that point
- Mimics real deployment conditions

**Metrics:** MAE, RMSE, MAPE, R²

**Baselines:**
1. Naïve Last-Week Repeat
2. 3-Week Moving Average
3. Linear Regression (No Lags)

## 5. Uncertainty Scenarios

Three scenarios from model residual distribution:

| Scenario | Derivation |
|----------|-----------|
| Lower sensitivity scenario | max(0, forecast − holdout RMSE) |
| Expected Case | Point forecast (median) |
| Upper sensitivity scenario | forecast + holdout RMSE |

**Important:** These are NOT calibrated prediction intervals.
They are heuristic ranges for contingency planning.

## 6. Spatial Exposure Allocation

City-level forecast cases allocated to zones via composite exposure index:

```
exposure_index = (
    density_weight × 0.35
    + facility_pressure_weight × 0.25
    + mobility_corridor_weight × 0.20
    + vulnerability_weight × 0.20
) × current_anomaly_adjustment
```

**Limitation:** This is a heuristic, not a validated spatial epidemiology model.
Phase 1 requires ward-level surveillance validation.

## 7. Operational Metrics

### Supply Depletion Horizon (SDH)
```
Dynamic Daily Demand = Baseline Daily Consumption × Growth Factor
SDH = Current Stock / Dynamic Daily Demand
```

### Projected Bed Load
```
Projected Bed Load = Occupied Beds + (Daily Surge Cases × Avg LOS)
Bed Deficit = max(0, Projected Bed Load - Dengue Bed Capacity)
```

### Experimental Planning-Priority Score
```
Planning Score = exposure × vulnerability × 200 + exposure × 80 + experimental_growth_score × (0.60 + vulnerability × 0.30), capped at 100
```

## 9. Model Selection Rationale

### Active model and historical GBR evidence

Random Forest is the active P1.2B model because it won the declared synthetic rolling-origin comparison. The following GBR discussion is retained as historical P0.4/P1.1 compatibility context:

1. **Best validation performance:** GBR achieved the lowest MAE (47.0) and RMSE (67.8)
   under chronological holdout on the synthetic demo dataset, outperforming both the
   naive baseline (MAE 149.0) and moving average baseline (MAE 223.7) by substantial
   margins (68% and 79% reduction in MAE respectively).

2. **Appropriate for tabular lag features:** The feature matrix produced by Phase 2
   feature engineering is a structured tabular dataset with 18 lag, rolling, and
   seasonality features. GBR is a well-validated method for this type of input.

3. **No novelty claim:** The model is not presented as a novel contribution. It is used
   as a practical, interpretable tabular forecasting method. The research contribution
   lies in the operational decision-support layer and spatial uncertainty propagation,
   not in the forecasting algorithm itself.

4. **Feature-importance policy:** no active feature-importance evidence is currently generated;
   allowing the dashboard to show which lag features are driving the forecast — a
   transparency requirement for health system decision support.

### Why were baselines included?

Baselines (naive last-known and 4-week moving average) serve two purposes:

1. **Credibility gate:** A model that cannot outperform a naive "last week continues"
   assumption provides no useful operational signal. Including baselines proves that
   the ML pipeline adds measurable value.

2. **Contextual anchor:** During low-volatility periods, the naive baseline performs
   acceptably. The gap between GBR and naive performance is largest during transition
   periods (surge onset, post-peak decline) — precisely when operational decisions
   are highest-stakes.

### Why time-based validation?

Random train/test splits are scientifically invalid for time series forecasting:
- If training data contains future observations (e.g. 2026 W12 in train, 2025 W40
  in test), the model effectively memorises future patterns.
- This produces artificially optimistic test metrics that collapse in deployment.

A strict chronological holdout (first 80% train, last 20% test, no shuffling) mirrors
real operational conditions where forecasts are always made into genuinely unseen
future periods.

### Why browser upload is currently validation-only

Full ML recalculation (feature engineering → governed Random Forest adoption → RF-bound sensitivity → directives)
requires Python, pandas, numpy, and scikit-learn. Running this in the browser
would require WebAssembly-compiled ML libraries, which is out of scope for this
research prototype.

The current browser upload interface:
- Validates CSV schema (column names and structure) client-side using FileReader
- Shows row counts and detected/missing columns immediately without server round-trips
- Downloads CSV templates matching the exact data contract
- Confirms readiness for the Python pipeline

In future deployment, validated CSV uploads would be forwarded to a queued backend
pipeline job (e.g. FastAPI endpoint + background Celery task), with JSON outputs
served back to the dashboard on completion.

## 11. Evidence, Profile, and Model Governance

P0.3C validates a versioned deployment profile, formula registry, and evidence registry before profiled input generation. The current evidence registry is intentionally empty. The Dhaka South profile is a deterministic synthetic capability demonstration at `benchmark_only`; it supplies the profile-driven maturity, demonstration, prohibited-claim, and simulated-notification wording used by generated metadata.

`observed_data_mode` reports whether the selected source classes are synthetic, real, or mixed. It is distinct from profile deployment maturity and does not establish local calibration, epidemiological validation, hospital approval, or institutional authorization. Each profiled run generates a model card with the actual library version, training period, single chronological holdout metrics, baseline models, and uncalibrated RMSE sensitivity limitation.

The open `TD-P03A-LEGACY-RISK-FIELDS` compatibility contract remains unchanged; legacy fields continue to be generated but are marked deprecated in the profile and model card.

## 12. Run-Specific Feature Diagnostics

P0.4 computes two diagnostics from the fitted chronological validation-model instance: native tree impurity importance and permutation importance on the unseen chronological holdout. Permutation importance is ranked primarily using `neg_mean_absolute_error`, 20 repeats, and random seed 42. Its values represent the holdout MAE change after permutation and may be negative; negative values are retained rather than clamped.

These diagnostics do not directly explain the separately fitted all-data forecast-model instance. They do not establish causality, biological mechanism, clinical importance, or stability across seasons. Correlated lag features may divide or mask importance, and synthetic benchmark relationships may influence rankings.
