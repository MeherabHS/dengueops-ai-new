# DengueOps AI — Assumptions & Limitations

## Preamble

Transparency about assumptions and limitations is a core design principle of
DengueOps AI. This document discloses all known simplifications, approximations,
and constraints embedded in the Phase 0 prototype.

---

## Data Assumptions

| Assumption | Detail | Phase 1 Resolution |
|-----------|--------|-------------------|
| Facility names use public anchors where available | Real government/public hospital names used as credible location anchors (DMCH, SSMC, Mugda MCH, NIBPS, Kamrangirchar 31-Bed) | Validated affiliation from health authority |
| General bed capacity is a public reference | `general_bed_capacity` for real anchors uses publicly known/official figures | Validated data from DGHS / DSCC |
| Dengue bed allocation is synthetic | `dengue_bed_capacity_demo` is a synthetic subset — does NOT claim to represent real dengue ward allocation | Validated dengue bed data from health authority |
| Dengue bed occupancy is synthetic | `occupied_dengue_beds_demo` is a synthetic demo figure only | Validated occupancy data from facility records |
| Facility inventory is synthetic | All NS1/RDT kit stock, IV fluid stock, and consumption rates are synthetic demo values | Validated inventory from DSCC/facility records |
| Validation uses aggregate/synthetic data | Backtest metrics are indicative, not peer-reviewed | Full validation on verified historical surveillance data |

---

## Methodology Assumptions

| Assumption | Detail | Impact if violated |
|-----------|--------|-------------------|
| Zone allocation is a spatial exposure heuristic | Not a validated spatial epidemiology model | Resources may be misallocated to/from zones |
| 2- and 4-week climate lags | Provisional candidate inputs; not locally selected or proven for Dhaka | Lag mismatch reduces forecast accuracy |
| Uncertainty bands are heuristic | Not statistically calibrated prediction intervals | Under/overestimation of scenario ranges |
| Bed load uses LOS approximation | Does not model severity mix, discharge variability, or referral patterns | Bed gap estimates may be inaccurate |
| Growth factor linearly scales demand | Assumes proportional supply-demand scaling | Non-linear surge dynamics not captured |

---

## Deployment Limitations

1. **No real-time data in Phase 0.** All data is static JSON. No live API, database, or surveillance feed.

2. **No authentication or access control.** Phase 0 is a public showcase prototype only.

3. **No automated alerts.** All directives require manual review and action.

4. **No multi-city support.** Phase 0 is scoped to Dhaka South only.

5. **No Bangla language interface.** Phase 0 is English only.

6. **System is advisory, not autonomous.** This is a fundamental design constraint, not a limitation to be resolved.

---

## Model Limitations

1. **Random Forest is not a sequence model and cannot extrapolate.** Temporal dependencies beyond the lag window are not captured, and predictions remain bounded by learned tree responses.

2. **MDI feature importances can be biased.** High-cardinality features may appear artificially important.

3. **No causal modelling.** Correlations between climate and dengue are associations, not causal pathways.

4. **Training data may not cover extreme surge events.** Out-of-distribution surge magnitudes may have higher prediction error.

---

## What This Means for Users

- Results are indicative planning aids, not precise operational forecasts.
- Current SDH triggers (critical 3 days; NS1 warning 7 days; IV-fluid warning 5 days), bed-deficit arithmetic, and planning tiers are governed prototype rules. The thresholds are not institution-approved.
- Directivesrequire professional judgement before any action is taken.
- Ground-truth validation is required before any real-world use.

## Data Input Limitations (Phase 0–5)

### Current data source assumptions

1. **Dashboard uses preloaded synthetic/demo data.** All CSV files under `data/`
   (`dengue_cases.csv`, `climate_data.csv`, `zones.json`, `facilities.json`,
   `inventory.json`) are generated by `analytics/generate_demo_data.py` from a
   parametric seasonal model with fixed random seed 42. They do not represent real
   DGHS, IEDCR, or DSCC surveillance records.

2. **JSON outputs are pre-computed and static during a session.** The dashboard
   reads from `data/forecast_output.json`, `data/validation_metrics.json`, and
   `data/directives.json` as pre-computed artifacts. The dashboard does not trigger
   model retraining on page load.

3. **Uploaded CSV support validates schema and previews structure only.** The
   CsvUploadPanel component in the dashboard checks column names and row counts
   client-side (FileReader API, no server required). It does not execute feature
   engineering, model training, or forecast generation in the browser. Those steps
   run through the Python analytics pipeline.

4. **Full retraining/reforecasting currently runs through local Python pipeline.**
   To recalculate with new data after uploading validated CSVs, the user must run:
   ```bash
   # Replace data/dengue_cases.csv and data/climate_data.csv with validated files, then:
   python analytics/feature_engineering.py
   python analytics/validation_backtest.py
   python analytics/forecast_model.py
   python analytics/uncertainty_engine.py
   python analytics/operational_engine.py
   # Refresh the dashboard to see updated outputs
   ```

5. **P1.4B connects upload only to authoritative validation.** The confirmed host is
   a persistent Ubuntu VPS with a long-lived Next.js Node server, Python, and local
   persistent disk. The Node-runtime route accepts bounded CSV uploads and invokes a
   validation-only Python entry point inside an isolated workspace. It does not use
   FastAPI, Celery, Redis, polling, or a forecasting job.

6. **A validated upload does not replace the current dashboard.** P1.4B does not
   train a model, compare candidates, generate uncertainty or preparedness outputs,
   create a completed run, or update a latest-run pointer. The bundled benchmark
   remains authoritative for Overview until a future complete run is validated and
   committed. Synthetic P1.3 coverage and the legacy 87/120/153 planning scenarios
   are not transferred to uploaded data.

7. **Runtime storage is separate from governed benchmark artifacts.** Uploads and
   canonical files live beneath the absolute `DENGUEOPS_RUNTIME_ROOT`; configuration
   that points this root at `data/` is rejected. Files default to 10 MiB each and the
   production Nginx reverse proxy must apply a corresponding request-body limit.

8. **P1.4C-2 is a point-only, policy-gated runtime.** Eligible uploads are processed
   in isolated staging and become visible only after immutable commit and atomic
   deployment-pointer replacement. The approved Random Forest is not claimed to be
   the best model for the upload.

9. **Uploaded uncertainty and preparedness remain unavailable.** No P1.3 range,
   nominal or historical coverage, RMSE fallback, 87/120/153 scenario, facility
   projection, inventory alert, or directive is inherited by an uploaded run.
   Dataset-specific temporal calibration and a governed planning-scenario policy
   are required before those outputs can be added.

10. **The runtime remains trusted/internal.** It assumes a persistent Ubuntu VPS,
    a dedicated unprivileged service user, systemd supervision, protected
    environment configuration, Nginx body/rate limits, and disk monitoring. Public
    execution requires authentication and authorization that are not in this phase.

## P0.3C Governance Limitations

- The Dhaka South deployment profile is `benchmark_only` and uses deterministic synthetic benchmark cases, climate, facilities, and inventory.
- The evidence registry is currently empty. No scientific evidence, local validation evidence, or institutional approval record is represented.
- Profile `data_mode` records governance maturity; `observed_data_mode` records source reality. Real or mixed input status alone would not mean locally calibrated or approved.
- The generated model card reports one chronological holdout and an uncalibrated RMSE sensitivity band, not deployment-grade validation or a probabilistic interval.
- Simulated preparedness notification wording describes a workflow demonstration; no message is sent to an operational recipient.
- `TD-P03A-LEGACY-RISK-FIELDS` remains open. Legacy fields are preserved for compatibility and must not be treated as the canonical contract.

### P0.4 explainability limitations

- P0.4 permutation diagnostics remain based on the 35-row chronological holdout. P1.1 separately aggregates native tree importance across rolling-origin estimators; permutation importance is not calculated on one-row folds because it would be statistically degenerate.
- Rolling-origin validation repeatedly evaluates unseen future periods, but current results are based on deterministic synthetic benchmark data and do not establish real-world Dhaka performance.
- The label-availability policy treats a target occurring at the forecast origin as available; real reporting publication delays and revision vintages are not modeled yet.
- Candidate comparison uses fixed configurations on deterministic synthetic data. Its winner is a technical demonstration choice, not evidence of real-world Dhaka superiority.
- Rolling folds and two-week targets overlap temporally, so paired error summaries are descriptive and do not support independence or significance claims.
- The P1.2A Random Forest winner is adopted under P1.2B only for the synthetic capability demonstration. P1.3 evaluates one empirical range using prior-only RF rolling residuals, but overlapping targets, temporal dependence, and model-selection reuse limit interpretation. It is not a prediction interval or a probability guarantee. The legacy RF RMSE triplet remains planning compatibility only.
- The explained estimator is the fitted validation-model instance, not the separate all-data forecast-model instance.
- Native tree importance may favor available split structures and may distribute importance unpredictably across correlated features.
- Permutation importance may be negative and may understate correlated features; negative values can reflect noise, instability, correlation, or holdout sampling variation.
- Rankings describe model behavior in the deterministic synthetic benchmark run and are not epidemiological or causal effect estimates.

## Runtime dataset assessment limitations

- Runtime assessment is restricted to the active synthetic-benchmark-compatible Dhaka South contract with exactly 173 labelled rows and 68 governed folds.
- The technical winner is not an adoption decision. Recommendation strength remains `not_available`, approval is disabled, and deployment state is unchanged.
- No forecast, uncertainty calibration, preparedness result, facility projection, alert, or operational directive is generated from assessment evidence.
- Assessment execution is intended for trusted/internal use until authentication, authorization, rate limiting, and production resource limits are deployed.

## Trusted internal one-run model-use limitations

- An `evidence_only` technical winner may inform a manual internal decision, but recommendation strength remains `not_available`; no automatic approval or deployment-wide adoption occurs.
- Decision identity is `trusted_internal_unverified`. The configured operator identifier is not authenticated and must not be described as institutional, clinical, or regulatory approval.
- Only governed learned candidates can produce an approved forecast. Naive baselines remain evaluation-only even when they are the technical winner.
- Each authorization is one-run and append-only: it is reserved at job creation and consumed only after a successful immutable run and latest-pointer update. Failures require controlled future recovery rather than automatic retry.
- Approved forecasts are point forecasts only. Bounds and coverage are null, P1.3 synthetic calibration is not reused, preparedness is unavailable, and no facility, inventory, alert, directive, or planning-scenario output is generated.
- Internal deployment requires HTTPS, an Nginx IP allowlist and rate limits, a server-held decision secret, and an unprivileged worker. Public use remains out of scope until proper authentication and authorization are implemented.
